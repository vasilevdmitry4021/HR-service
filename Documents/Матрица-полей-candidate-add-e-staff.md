# Матрица заполнения полей `candidate/add` (e-staff)

Источник контракта: `Documents/методы estaff.md` (EStaff_Candidate_API, обновление 27.01.2026).

## Минимально необходимое для создания кандидата

- **firstname** — обязательное поле кандидата (*1 в документации).
- **Контакт** — обязательно хотя бы одно из: мобильный телефон или e-mail (*2).
- **vacancy.id** — обязательный блок вакансии при привязке к вакансии в том же запросе.

Поле **lastname** в примере JSON присутствует; при отсутствии в резюме допускается пустое значение только если e-staff это принимает — на практике заполняем из HeadHunter или заглушки по настройкам сервера.

---

## Группа A. Простое копирование (строки и числа без справочника)

| Поле e-staff | Источник в резюме (HH) | Правило |
|--------------|------------------------|---------|
| lastname, firstname, middlename | `_raw.last_name`, `_raw.first_name`, `_raw.middle_name` | Обрезка пробелов; при пустых значениях — политика `estaff_fio_placeholder_*` |
| mobile_phone, email | `_raw.contact[]` | Разбор типов контактов HH |
| home_phone, email2, skype | `_raw` при наличии | По мере появления в нормализованном сырье |
| desired_position_name | `title` | Строка должности |
| salary | `salary.amount` | Число, разбор строки при необходимости |
| birth_date | `_raw.birth_date` | Если доступно в API HH — ISO |
| time_zone, children_num | редко в HH | Не заполнять без данных |
| inet_uid | `hh_resume_id` | Внешний идентификатор резюме |
| city_name | `area` (название города/региона) | Текст; по документации может участвовать в сопоставлении с `locations` в e-staff |

---

## Группа B. Фиксированные перечисления (таблица на сервере, не `get_voc`)

| Поле | Допустимые значения (документация) | Сопоставление с HH |
|------|-----------------------------------|---------------------|
| gender_id | 0 — муж., 1 — жен. | `_raw.gender`: `male` → 0, `female` → 1; иначе поле не отправлять |
| educ_type_id | 1 среднее, 2 среднее проф., 3 неоконч. высшее, 4 высшее | Эвристика по текстам уровня в `education` / `primary` блоках |
| marital_status_id | 1–4 (семейное положение) | Только при явном согласованном маппинге с HH; иначе не отправлять |

---

## Группа C. Справочники через `POST /api/base/get_voc`

Идентификатор справочника передаётся в теле `{"voc":{"id":"<код>"}}`. Ответ: массив элементов `{id, name}` под динамическим ключом (не `success`).

| Поле кандидата | Справочник (id в запросе `get_voc`) | Примечание |
|----------------|-------------------------------------|------------|
| country_id | `countries` | Если из резюме известна страна как текст |
| location_id | `locations` | Регион; документация: игнорируется при использовании только `city_name` в части сценариев — при неоднозначности не подставлять id |
| metro_station_id | `metro_stations` | При наличии текста/кода станции в данных |
| job_search_location_id[] | `locations` | Список регионов поиска |
| skills[].type_id | `skill_types` | Каждый навык из списка — отдельное разрешение |
| entrance_type_id | `candidate_entrance_types` | Опционально, бизнес-правило |
| source_id | `candidate_sources` | Опционально |
| state_id | `candidate_states` | Опционально |
| attachments[].type_id | `card_attachment_types` | Для вложений |

**Правило:** обязательные для API поля не заполнять «угадыванием» id; необязательные при отсутствии точного/однозначного совпадения — пропускать и фиксировать предупреждение в журнале и в результате выгрузки.

## Группа D. Составные блоки

| Блок | Поля | Источник HH |
|------|------|-------------|
| prev_educations[] | end_year, org_name, department_name, speciality_name | `education.primary` / `additional` / … |
| prev_jobs[] | start_year, start_month, end_*, org_name, org_location_name, position_name, comment | `experience[]` в `_raw` |
| skills[] | type_id, level_id, comment | `skill_set` + разрешение `type_id` через справочник |
| passport | number, issue_date, issue_org | При появлении в данных |
| csd | произвольные ключи | Доп. поля карточки — только по согласованной схеме |
| attachments[] | type_id, content_type, данные файла | По согласованному процессу |

---

## Поведение при ошибках разрешения

1. **Нет совпадения** — необязательное поле не включается в тело; предупреждение в результат подготовки.
2. **Несколько равнозначных совпадений** — поле не заполняется; предупреждение «неоднозначно».
3. **Сбой `get_voc`** — техническая ошибка: этап «vocabulary»; часть полей может быть пропущена с предупреждениями или выгрузка прервётся в зависимости от критичности (текущая реализация: не блокирует обязательный минимум, логирует предупреждения).

---

## Резерв: поле `candidate` с клиента

Передача готового объекта `candidate` с клиента остаётся вспомогательным путём: сервер не подменяет LLM-подготовку в основном сценарии; при передаче с клиента разрешение справочников на сервере может быть пропущено (фиксируется предупреждением).
