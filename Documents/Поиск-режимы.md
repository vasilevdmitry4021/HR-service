# Режимы поиска и пост-фильтрации

## Обзор пайплайна поиска

После получения результатов от API HeadHunter/мока применяется:

1. **Пост-фильтрация** по точным числовым границам из parsed (если включена)
2. **Эвристическая оценка** (relevance) для каждого резюме
3. **Сортировка** по баллу (и strict_match в режиме demote)
4. **Отбор топ-K** для LLM-анализа (только кандидаты с баллом выше порогов)
5. **LLM-анализ** — batch или по одному, с fallback при сбое парсинга

## Режимы пост-фильтрации (STRICT_FILTER_MODE)


| Режим    | Поведение                                                                                                         |
| -------- | ----------------------------------------------------------------------------------------------------------------- |
| `hide`   | Кандидаты, не прошедшие точные границы (experience_years_min, age_min, age_max, salary_from), убираются из выдачи |
| `demote` | Не прошедшие остаются в выдаче, но помечаются `strict_match=false` и отодвигаются в конец списка                  |


Параметр `STRICT_NUMERIC_FILTERS` включает или отключает пост-фильтрацию целиком.

## LLM на списке

- `LLM_SEARCH_MAX_CANDIDATES` — максимум резюме на странице, для которых вызывается LLM
- `LLM_SEARCH_BATCH_SIZE` — при 1 каждый резюме анализируется отдельно; при >1 — пакетные запросы с fallback на одиночные при сбое
- `LLM_ANALYSIS_HARD_FLOOR` — ниже этого эвристического балла LLM не вызывается даже для кандидатов в топ-K
