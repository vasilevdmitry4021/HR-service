## 3. Обработка и структурирование запроса

| № | Требование | Статус | Комментарий |

| 3.1 | Анализ текстового запроса | Сделано | Разбор через правила (режим имитации языковой модели) или через подключённую языковую модель (зависит от настроек сервера). |
| 3.2 | Автоматическое выделение критериев поиска | Сделано | Результат разбора возвращается с бэкенда; на экране отображаются теги/превью распознанных параметров (где предусмотрено интерфейсом). |
| 3.3 | Структурирование параметров для API HH | Частично | См. детализацию ниже. |

### Детализация по подпунктам 3.3 (критерии из требований)

| Критерий | Статус | Комментарий |
|----------|--------|-------------|
| Регион | Частично | Есть выбор региона в панели фильтров (ограниченный список идентификаторов) и подстановка из текста при совпадении с известными названиями. Полный справочник регионов HH в интерфейсе не подключён. |
| Возраст | Частично | В панели заданы поля «возраст от/до». Из текста частично подставляется верхняя граница (например, распознавание «до N лет»). Не все варианты формулировок в возрасте могут отражаться в панели автоматически. |
| Пол | Сделано | Панель фильтров и перенос из разобранного текста (в т.ч. фоновое обновление панели при наборе запроса). |
| Профессиональные навыки | Частично | Из текста навыки извлекаются и участвуют в оценке релевантности и в имитации базы. В запрос к реальному API HH как отдельные идентификаторы навыков из панели они не подставляются автоматически из разбора; в панели нет выбора навыков из справочника HH. |
| Отрасль | Частично | В тестовом разборе текста отраслевые маркеры распознаются ограниченно. В панели фильтров нет выбора отрасли по справочнику HH; слияние с параметром `industry` API из разбора не выполняется так же полно, как для региона или пола. |
| Должность | Частично | В разборе есть ключевые слова по должности; в API-параметр «профессиональная роль» HH из текста и из панели в текущем виде не выводится в том объёме, как ожидается в примере из требований (Java-разработчик / backend и т.д.). |
| Опыт работы | Сделано | Категории опыта в панели соответствуют типичным фильтрам HH; из текста минимальный стаж маппится в категорию опыта на бэкенде и частично подставляется в панель. |

### Примечание из «Требования.md» (явные фильтры + подстановка «на лету»)

| Ожидание из примечания | Статус | Комментарий |
|------------------------|--------|-------------|
| Чекбоксы и списки, явно соответствующие фильтрам HH | Частично | Реализованы основные поля: регион, опыт, пол, возраст, зарплата, валюта. Нет полноценных списков ролей, отраслей и навыков HH с идентификаторами API. |
| Подстановка фильтров из текста «на лету» | Частично | При наборе запроса выполняется отложенный запрос разбора и аккуратное заполнение **пустых** полей панели (регион, опыт, пол, верхняя граница возраста). Должность, отрасль и навыки в панель так не синхронизируются. |
| Одному критерию — несколько значений в фильтрах | Не сделано | В интерфейсе в основном одиночный выбор (например, один регион); задел на бэкенде для списков есть не для всех полей в связке с UI. |

**Открытые задачи (3.x):**

- [ ] Расширить панель фильтров: отрасль и профессиональная роль по справочникам HH (с загрузкой или кэшем справочников).
- [ ] Добавить выбор навыков (идентификаторы HH), согласовать с разбором текста (маппинг названий → id).
- [ ] Дополнить `partialFiltersFromParsedParams` и слияние с API: должность → `professional_role`, отрасль → `industry`, при необходимости — навыки → `skill`.
- [ ] Расширить список регионов (или поиск по справочнику HH) вместо короткого фиксированного перечня.
- [ ] Поддержать несколько значений там, где это допускает API HH и логика требований (и отразить в UI).
- [ ] При отключённой имитации языковой модели: проверить и настроить промпт/схему ответа LLM под все перечисленные в ТЗ поля.
- [ ] Учесть флаг расширенных фильтров в конфигурации (`feature_advanced_filters`), если планируется вынос «тяжёлых» фильтров за отдельный режим.
