# Вызов `get_voc`, поиск по справочникам и кэш

## HTTP-вызов

- Метод: `POST` к пути из настройки `ESTAFF_GET_VOC_PATH` (по умолчанию `/api/base/get_voc`).
- Тело: `{"voc":{"id":"<код_справочника>"}}`, заголовок `Authorization: Bearer <token>`.
- Реализация: `backend/app/services/estaff_client.py` — функция `fetch_get_voc`, обработка сетевых и HTTP-ошибок по образцу других вызовов e-staff, повторы при 5xx.

## Разбор ответа

Ответ содержит массив элементов под динамическим ключом (не `success`). Функция `parse_get_voc_response` в `estaff_vocabulary.py` извлекает список словарей с полями `id` и `name`.

## Правила поиска идентификатора

1. Нормализация строки: нижний регистр, схлопывание пробелов, снятие типовых префиксов («г.», «город»).
2. Сначала **точное** совпадение нормализованного `name` с запросом.
3. Если включено **префиксное** сопоставление (`allow_prefix`): одно однозначное совпадение по началу строки (в обе стороны).
4. Ноль совпадений — поле не заполняется, причина `not_found`.
5. Несколько равнозначных совпадений — не выбирать автоматически, причина `ambiguous`.
6. Нечёткий поиск по подстроке намеренно **не** используется, чтобы снизить риск неверного id.

## Кэш

- Модуль `estaff_vocabulary.py`: обёртка `get_vocabulary_items_cached` поверх `TTLCache` из `ttl_cache.py`.
- Ключ кэша: хост/стенд, укороченный хэш токена (сам токен в ключ не кладётся), идентификатор справочника.
- Параметры: `ESTAFF_VOC_CACHE_TTL_SECONDS` (по умолчанию 600 с), `ESTAFF_VOC_CACHE_MAX_ENTRIES`.

## Связь с подготовкой кандидата

Единая подготовка тела `candidate` — `prepare_candidate_payload_for_export` в `estaff_candidate_prepare.py`: при необходимости запрашивает `locations` и `skill_types`, записывает предупреждения при сбоях справочника или неоднозначности, не подставляет выдуманные id.
