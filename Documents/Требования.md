1. Управление доступом к API

1.1. Система должна предоставлять возможность ввода и сохранения API-ключа (ключа доступа) от личного кабинета HH.

1.2. Система должна использовать сохранённый API-ключ для выполнения запросов к API HH.

2. Получение резюме из нескольких источников

2.1. Система должна обеспечивать возможность получения резюме не только с HeadHunter, но и из дополнительных площадок, согласованных с заказчиком, в том числе: SuperJob, Rabota.ru, Avito Работа, Хабр Карьера.

2.2. Для каждой подключаемой площадки должна предусматриваться настройка доступа (ключи API, учётные данные личного кабинета работодателя или иной предусмотренный площадкой и договором способ подключения).

2.3. Пользователь или настройки развёртывания должны позволять включать и отключать источники, а также использовать при поиске и отборе кандидатов данные из одного или нескольких подключённых источников в соответствии с возможностями каждой площадки.

3. Формирование пользовательского запроса

3.1. Система должна предоставлять интерфейс для ввода текстового запроса на поиск кандидатов в свободной форме.

3.2. Запрос должен поддерживать описание требований к кандидату (например: опыт работы, технологии, отрасль, должность и т.д.).

Пример запроса:Мне нужен сотрудник с опытом разработки на Java в ритейле.

4. Обработка и структурирование запроса

4.1. Система должна анализировать текстовый запрос пользователя.

4.2. Система должна автоматически выделять критерии поиска кандидатов.

4.3. Система должна определять и структурировать параметры запроса для обращения к API HH, включая при наличии в запросе:

- Регион;
- Возраст;
- Пол;
- Профессиональные навыки;
- Отрасль;
- Должность;
- Опыт работы.

Примечание: 
Я предлагаю сделать над промптом обычные чек-боксы / списки параметров, в явном виде.
Они соответствуют фильтрам в HeadHunter, чтобы не нужно было это писать в промпт.  

Например, я могу просто выбрать из списка регион, пол, написать диапазон возврастов. А потом уже уточнить.

И обратно,  при заполнении промпта, если распознается связанный с фильтром критерий, то  они будут ?на лету заполняться?.

Т.е. если я пишу ?Найди мне мужчину  с опытом разработки на java в ритейле?, то сразу проставляются фильтры на экране ?пол? ?должность=разработчик JAV A” или “должность = разработчик backend?.    У нас одному критерию могут соответствовать несколько значений в фильтрах.

5. Интеграция с e-staff

5.1. Система должна поддерживать интеграцию с сервисом e-staff для передачи (выгрузки) данных по найденным и отобранным резюме в учёт подбора заказчика.

5.2. Система должна предусматривать настройку подключения к e-staff (учётные данные, ключ API или иной предусмотренный e-staff способ аутентификации — по документации выбранной схемы интеграции).

5.3. Пользователь должен иметь возможность инициировать выгрузку в e-staff для одного или нескольких резюме, отобранных в системе (в том числе из избранного), с передачей согласованного набора полей, допускаемого политикой HeadHunter и настройками e-staff.

5.4. Система должна фиксировать результат операции выгрузки (успех, ошибка, идентификатор кандидата в e-staff при наличии в ответе сервиса) для контроля и повторных попыток при сбоях.
