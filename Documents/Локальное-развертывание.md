# Инструкция по локальному развёртыванию HR Service

Документ описывает развёртывание приложения на рабочей машине разработчика или тестовом ПК (не продакшен-сервер). Стек: **FastAPI**, **PostgreSQL**, **Next.js 14** (App Router).

---

## 1. Что понадобится

- **Git** — клонирование репозитория.
- **Docker Desktop** (или совместимый Docker + Compose) — если выбираете вариант с контейнерами **или** только для PostgreSQL/Redis.
- Либо **без Docker:**
  - **Python 3.11+** (версию ориентируйтесь на `backend/requirements.txt` и CI),
  - **Node.js** (LTS) и npm,
  - **PostgreSQL 16** (удобно совпадать с образом в `docker-compose.yml`).

---

## 2. Файлы окружения

Перед первым запуском скопируйте примеры:

| Откуда | Куда | Назначение |
|--------|------|------------|
| `backend/.env.example` | `backend/.env` | Backend: `DATABASE_URL`, секреты JWT, интеграции (HH, OAuth, e-staff и т.д.) |
| `.env.example` | `.env` (корень репозитория) | Переменные для **Docker Compose** (порты, БД, часть флагов) |
| `frontend/.env.example` | `frontend/.env.local` | URL API для браузера (`NEXT_PUBLIC_API_URL`) |

Подробные комментарии к переменным — внутри соответствующих `.env.example`.

**Важно для запуска backend вне Docker:** в `backend/.env` укажите `DATABASE_URL` на ваш локальный PostgreSQL, например:

`postgresql://hr:hr@localhost:5432/hr_service`

Создайте базу и пользователя так же, как задано в URL (или измените URL под свою схему).

## 3. Вариант A: всё через Docker Compose

Из **корня** репозитория:

```bash
docker compose up --build
```

При старте сервиса `backend` обычно выполняется `alembic upgrade head`, затем поднимается API. Если схема БД не применилась:

```bash
docker compose exec backend alembic upgrade head
```

**Адреса по умолчанию:**

- API: [http://localhost:8000](http://localhost:8000) — `GET /health`, REST: префикс `/api/v1`
- Веб: [http://localhost:3000](http://localhost:3000)

Порты можно переопределить в корневом `.env` (см. `.env.example`: `POSTGRES_PORT`, `BACKEND_PORT`, `FRONTEND_PORT`).

---

## 4. Вариант B: backend и frontend на хосте (без контейнеров приложения)

### 4.1. База данных

Поднимите PostgreSQL (отдельно или только сервис `db` из Compose: `docker compose up -d db`). Убедитесь, что `DATABASE_URL` в `backend/.env` указывает на доступную базу.

### 4.2. Backend

В каталоге `backend`:

```bash
python -m venv .venv
# Windows: .venv\Scripts\activate
# Linux/macOS: source .venv/bin/activate
pip install -r requirements.txt
alembic upgrade head
uvicorn app.main:app --reload --host 0.0.0.0 --port 8000
```

### 4.3. Frontend

В каталоге `frontend`:

```bash
npm install
npm run dev
```

В `frontend/.env.local` задайте `NEXT_PUBLIC_API_URL` на URL вашего API (по умолчанию `http://localhost:8000`).

## 5. Первый суперпользователь (super-admin)

Флаг супер-администратора хранится **только в БД** (`users.is_super_admin`). Автоматически при пустой базе пользователь **не создаётся**. Назначение через email в конфиге не предусмотрено (намеренно, из соображений безопасности при открытой регистрации).

Рекомендуемый порядок на **новой** машине:

1. Применить миграции: `alembic upgrade head` (или дождаться старта `backend` в Docker).
2. Выполнить создание супер-админа (см. ниже).
3. Войти в UI этим email и паролем; при необходимости проверить `GET /api/v1/auth/me` — в ответе должно быть `is_super_admin: true`.
4. После этого имеет смысл открывать внешний доступ к приложению (если планируется).

### 5.1. Команда `create-super-admin`

Создаёт пользователя (если его ещё нет), задаёт пароль и выставляет `is_super_admin=true`. Если пользователь с таким email уже есть — при необходимости обновит флаг и пароль.

**Локальный Python** (из каталога `backend`, активированное venv):

```bash
cd backend
python -m app.cli.admin_bootstrap create-super-admin --email admin@example.com --password "StrongPass123!"
```

**Через контейнер backend** (из корня репозитория):

```bash
docker compose exec backend python -m app.cli.admin_bootstrap create-super-admin --email admin@example.com --password "StrongPass123!"
```

Подставьте свой email и надёжный пароль. Минимальные требования к паролю при регистрации через API см. в коде/документации API; для CLI используйте пароль достаточной длины и сложности.

### 5.2. Дополнительные команды CLI

| Команда | Назначение |
|---------|------------|
| `grant-super-admin --email ...` | Выдать super-admin **существующему** пользователю (например, после самостоятельной регистрации на `/register`) |
| `revoke-super-admin --email ...` | Снять super-admin; последнего супер-админа снять нельзя (защита в CLI) |

Запуск: тот же модуль `python -m app.cli.admin_bootstrap` с подкомандой.

## 6. Проверка работоспособности

- `GET http://localhost:8000/health` — ответ без ошибки.
- Открыть [http://localhost:3000/login](http://localhost:3000/login), войти учётной записью супер-админа.
- Опционально: с токеном авторизации вызвать `GET /api/v1/auth/me` и убедиться, что `is_super_admin` соответствует ожиданиям.

## 7. Экспорт PDF резюме (опционально)

В `backend/.env`: `FEATURE_PDF_EXPORT=true`  
В `frontend/.env.local`: `NEXT_PUBLIC_FEATURE_PDF_EXPORT=true`  

После перезапуска на карточке кандидата появится действие «Скачать PDF» (`GET /api/v1/candidates/{id}/pdf`).

---

## 8. Частые проблемы

- **Миграции не применились** — выполните вручную `alembic upgrade head` (в контейнере или локально с тем же `DATABASE_URL`).
- **Frontend не видит API** — проверьте `NEXT_PUBLIC_API_URL` и CORS/доступность порта 8000 с машины, где открыт браузер.
- **Другая машина в сети** — подставьте IP хоста вместо `localhost` в `NEXT_PUBLIC_API_URL` и в настройках OAuth/redirect URI, если используете внешние провайдеры.
