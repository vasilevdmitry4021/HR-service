# Сравнение подходов к подключению LLM в коммерческом продукте

## Подход 1: Запросы идут через ваш ID (единый ключ продавца)

Вы оформляете доступ к YandexGPT / GigaChat на своё юрлицо, ключи хранятся на вашем сервере, покупатели используют модель «под вашим капотом».

### Преимущества


| Аспект                      | Описание                                                                                 |
| --------------------------- | ---------------------------------------------------------------------------------------- |
| Простой онбординг           | Покупателю не нужно разбираться с облаком Яндекса или Сбера — всё работает «из коробки». |
| Единое качество сервиса     | Одна модель, одна версия, предсказуемое поведение для всех клиентов.                     |
| Контроль расходов и лимитов | Вы видите общий расход, можете устанавливать квоты на пользователя/организацию.          |
| Гибкость ценообразования    | Можно продавать «пакеты запросов» или безлимит в тарифе — клиент не думает про токены.   |
| Защита ключа                | Ключ не уходит к покупателю, меньше риск утечки.                                         |


### Недостатки


| Аспект                      | Описание                                                                                                    |
| --------------------------- | ----------------------------------------------------------------------------------------------------------- |
| Финансовый риск             | Если один клиент резко увеличит нагрузку, платите вы; нужно закладывать лимиты и защиту от злоупотреблений. |
| Операционная нагрузка       | Ответственность за uptime, мониторинг расхода, обновление токенов (OAuth для GigaChat).                     |
| Зависимость клиентов от вас | Если вы прекратите работу или не оплатите облако, модель перестанет работать у всех.                        |
| Требования к инфраструктуре | Нужен свой бэкенд или прокси-слой; нельзя просто раздать исходники с ключом.                                |
| Договорные ограничения      | Нужно проверять, разрешает ли оферта провайдера перепродажу/встраивание в коммерческий SaaS.                |

## Подход 2: Запросы идут через ID заказчика (ключ покупателя)

Вы продаёте ПО, покупатель сам оформляет доступ к YandexGPT / GigaChat и подключает свои учётные данные в настройках.

### Преимущества


| Аспект                                       | Описание                                                                                       |
| -------------------------------------------- | ---------------------------------------------------------------------------------------------- |
| Нет расходов на токены                       | За вызовы модели платит покупатель — ваш доход не «съедается» расходами на ИИ.                 |
| Масштабируемость без риска                   | Сколько бы клиентов ни было, ваш биллинг с облаком не растёт.                                  |
| Прозрачность для корпоративных клиентов      | Крупные компании предпочитают «данные идут под их договор», проще согласовать с ИБ и юристами. |
| Подходит для продажи исходников / on-premise | Ключ не зашит в код, можно передавать репозиторий.                                             |
| Меньше операционной нагрузки                 | Обновление токенов, мониторинг лимитов — забота покупателя.                                    |


### Недостатки


| Аспект                               | Описание                                                                                                                                      |
| ------------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------- |
| Сложный онбординг                    | Покупателю нужно: зарегистрироваться в облаке, оформить биллинг, получить ключ, настроить OAuth (GigaChat). Требуется подробная документация. |
| Разнородность окружений              | У разных клиентов могут быть разные тарифы, лимиты, версии моделей — сложнее воспроизводить баги.                                             |
| Часть поддержки «не про ваш продукт» | Обращения вида «не работает ключ» или «кончились токены» формально не ваша зона, но клиент всё равно пишет вам.                               |
| Риск некорректной настройки          | Если покупатель неправильно указал ключ или endpoint, продукт «не работает» — первое впечатление страдает.                                    |
| Ограничения провайдера               | Не все тарифы у Яндекса/Сбера доступны физлицам или малым ИП; покупателю может понадобиться юрлицо.                                           |

## Рекомендация

- **Для SaaS / облачного продукта** — начните с **подхода 1** (ваш ключ), чтобы упростить продажи и онбординг; позже добавьте опцию «свой ключ» для крупных клиентов.
- **Для продажи исходников / коробочного решения** — используйте **подход 2** (ключ покупателя) с подробной инструкцией по настройке.
- **Гибридный вариант** — в архитектуре заложите возможность указать любой endpoint и ключ; по умолчанию работает ваш (для SaaS-версии), но можно переключить на клиентский.
