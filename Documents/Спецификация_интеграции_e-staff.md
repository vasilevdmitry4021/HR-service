## 1. Цель и границы

### 1.1. Цель

Обеспечить передачу данных по отобранным резюме из HR-сервиса в учёт подбора заказчика в e-staff, с настраиваемым подключением и фиксацией результатов выгрузки.

### 1.2. Источники выгрузки

| Источник | Требование |
|----------|------------|
| Избранное | Одиночная и массовая выгрузка |
| Результаты поиска | Выгрузка из карточки кандидата в списке |
| Страница кандидата | Выгрузка с детальной карточки |

### 1.3. Учёт выгрузок

Для каждой операции выгрузки сохраняется запись в истории: статус, идентификатор кандидата в e-staff (если вернул API), сообщение об ошибке, время. Допускается несколько попыток для одного `hh_resume_id` (новая запись или обновление — по решению реализации; в плане заложена отдельная таблица записей).

### 1.4. Вне границ первой версии

- Реализация конкретных URL и тел методов e-staff задаётся по официальной OpenAPI в интерфейсе e-staff заказчика; в коде — конфигурируемый базовый адрес от `server_name`.
- Согласование полного набора полей резюме с политикой HeadHunter и настройками e-staff — ответственность заказчика; система передаёт согласованный минимальный набор (ФИО, контакты при наличии, привязка к вакансии e-staff при указании).

## 3. Данные и хранение

### 3.1. Таблица `estaff_exports`

| Поле | Тип | Обязательность | Описание |
|------|-----|----------------|----------|
| id | UUID | Да | Первичный ключ |
| user_id | UUID, FK → users | Да | Владелец записи |
| hh_resume_id | Строка | Да | Идентификатор резюме в HeadHunter |
| estaff_candidate_id | Строка | Нет | Идентификатор кандидата в e-staff после успеха |
| estaff_vacancy_id | Строка | Нет | Вакансия в e-staff, если передана |
| status | Перечисление | Да | `pending`, `success`, `error` |
| error_message | Текст | Нет | Текст ошибки при `error` |
| exported_at | Дата-время (с часовым поясом) | Нет | Момент успешной выгрузки |
| created_at | Дата-время (с часовым поясом) | Да | Создание записи |

Миграция: `backend/alembic/versions/YYYYMMDD_NNNN_estaff_exports.py`.  
Модель: `backend/app/models/estaff_export.py`.

### 3.2. Учётные данные e-staff

Хранение в таблице `system_settings`, ключ `estaff_credentials`, значение — зашифрованный JSON (как для HeadHunter): поля `server_name`, `api_token`.

Функция чтения: `get_estaff_credentials_from_db(db)` в `backend/app/config.py`.

## 4. Серверная часть (API и сервис)

### 4.1. Маршруты (префикс `/api/v1`, тег `estaff`)

| Метод | Путь | Назначение |
|-------|------|------------|
| GET | `/estaff/credentials` | Статус настройки (например, `configured: bool`), без раскрытия секрета |
| PUT | `/estaff/credentials` | Сохранение `server_name` и `api_token` (шифрование) |
| POST | `/estaff/export` | Выгрузка одного или пакета резюме |
| GET | `/estaff/exports` | История выгрузок текущего пользователя (пагинация по желанию) |
| GET | `/estaff/exports/{hh_resume_id}` | Актуальный или последний статус выгрузки по резюме |

Все маршруты — только для аутентифицированного пользователя (как остальные защищённые API).

Реализация: `backend/app/api/estaff.py`, подключение в `backend/app/main.py`.

### 4.2. Схемы Pydantic

Файл `backend/app/schemas/estaff.py`:

- Вход сохранения настроек: `server_name`, `api_token`.
- Ответ статуса настроек: признак наличия сохранённых данных.
- Элемент пакетной выгрузки: `hh_resume_id`, данные кандидата для e-staff (структура по OpenAPI), опционально `vacancy_id`.
- Ответ по записи выгрузки: идентификатор записи, статус, `estaff_candidate_id`, время и пр. по необходимости.

### 4.3. Клиент e-staff

Файл `backend/app/services/estaff_client.py`:

- Базовый URL формируется из `server_name` (уточнить у заказчика фактический шаблон хоста e-staff; в плане указан вид `https://{server_name}.e-staff.ru/api`).
- Заголовок: `Authorization: Token {api_token}`.
- Функции (имена и сигнатуры по плану): добавление кандидата, поиск кандидата, смена статуса — при необходимости для сценария «без дублей» или пост-обработки.
- Ошибки HTTP: единый класс исключения с кодом и текстом для записи в `error_message` и ответа клиенту.

При отсутствии доступа к реальному API — опциональный режим заглушки по аналогии с `feature_use_mock_hh` (не входит в обязательный объём; при внедрении указать в `.env`).

### 4.4. Логика POST `/estaff/export`

1. Проверить наличие учётных данных e-staff.
2. Для каждого элемента: при необходимости получить полное резюме через существующий клиент HeadHunter (токен пользователя).
3. Сопоставить поля с контрактом e-staff (маппинг в сервисе или в теле запроса от фронта — зафиксировать при реализации; предпочтительно централизовать на бэкенде).
4. Вызвать API e-staff, записать строку в `estaff_exports` со статусом `success` или `error`, заполнить `estaff_candidate_id` из ответа при успехе.

## 6. Нефункциональные требования

- Секреты не возвращать в ответах API и не логировать в открытом виде.
- Ошибки e-staff и сети отдавать пользователю в обобщённом виде, детали — в `error_message` в БД для поддержки.
- Соблюдать существующие стили интерфейса и паттерн `apiFetch` / авторизации.

---

## 7. Критерии приёмки

1. Администратор или пользователь может сохранить домен и токен e-staff; после сохранения статус «настроено» отображается без утечки токена.
2. С любой из трёх точек (избранное, поиск, карточка кандидата) можно инициировать выгрузку; из избранного доступна массовая выгрузка выбранных.
3. После выгрузки в БД есть запись с корректным `status`; при успехе при наличии в ответе e-staff заполнен `estaff_candidate_id`.
4. Пользователь видит признак успеха или ошибки; при ошибке возможна повторная попытка (новая запись или обновление — согласно выбранной реализации в коде).

## 10. Рекомендации по уточнению

Ниже приведены аспекты, которые рекомендуется детализировать перед или в процессе реализации.

### 10.1. Индексы таблицы `estaff_exports`

Для эффективных выборок необходимо предусмотреть индексы:

| Поле | Обоснование |
|------|-------------|
| user_id | Выборка истории выгрузок пользователя |
| hh_resume_id | Проверка статуса по конкретному резюме |
| created_at | Сортировка по дате при пагинации |

### 10.2. Обработка массовой выгрузки

При пакетной выгрузке нескольких резюме уточнить:

- Последовательная или параллельная обработка запросов к API e-staff.
- Учёт ограничений API e-staff (rate limits) — при необходимости добавить задержку между запросами.
- Поведение при частичном успехе: если часть записей выгружена, а часть завершилась с ошибкой — отображать сводку пользователю.

### 10.3. Привязка к вакансии e-staff

Поле `estaff_vacancy_id` предусмотрено в модели, однако не описан механизм выбора вакансии:

- Требуется ли API-метод для получения списка открытых вакансий из e-staff?
- Интерфейс: выпадающий список или поле для ввода идентификатора?
- Является ли привязка к вакансии обязательной или опциональной?

### 10.4. Проверка дублей

Определить логику при повторной выгрузке того же резюме:

- Показывать предупреждение, если резюме уже было успешно выгружено ранее.
- Уточнить поведение API e-staff: создаётся дубликат или обновляется существующая запись?
- При необходимости добавить опцию «Обновить существующего кандидата» vs «Создать нового».

### 10.5. Валидация данных перед выгрузкой

Описать проверку полноты данных:

- Минимальный набор полей, обязательных для e-staff (ФИО, контакт).
- Поведение при отсутствии обязательных полей: отклонение с сообщением или выгрузка с неполными данными.
- Валидация формата данных (телефон, email).

### 10.6. Разграничение прав доступа

Уточнить ролевую модель:

| Операция | Доступ |
|----------|--------|
| Сохранение credentials | Администратор или любой пользователь? |
| Выгрузка резюме | Любой аутентифицированный пользователь |
| Просмотр истории | Только своя история или общая? |

### 10.7. Таймауты и повторные попытки

Для клиента e-staff предусмотреть:

- Таймаут HTTP-запроса (рекомендация: 30 секунд).
- Стратегия retry при временных сбоях (5xx, таймаут): количество попыток, интервал между ними.
- Логирование неуспешных попыток для диагностики.

### 10.10. Пагинация истории выгрузок

Для эндпоинта `GET /estaff/exports` определить:

- Размер страницы по умолчанию (рекомендация: 20).
- Максимальный размер страницы (рекомендация: 100).
- Порядок сортировки: по `created_at` убыванию (новые первыми).
- Параметры запроса: `page`, `per_page` или `offset`, `limit`.

### 10.11. Ограничения пакетной выгрузки

Определить лимиты для предотвращения перегрузки системы и API e-staff:

| Параметр | Рекомендация | Обоснование |
|----------|--------------|-------------|
| Максимум резюме в одном запросе | 50 | Ограничение времени выполнения запроса |
| Минимальный интервал между запросами | 100 мс | Соблюдение rate limits e-staff |
| Максимум параллельных запросов к e-staff | 3 | Баланс скорости и нагрузки |

При превышении лимита — возвращать ошибку с указанием максимально допустимого количества.

### 10.12. Асинхронная обработка массовой выгрузки

При выгрузке большого количества резюме (более 10) рекомендуется асинхронная обработка:

**Вариант A (синхронный, простой):**
- Обработка в рамках HTTP-запроса.
- Подходит для небольших пакетов (до 10 записей).
- Таймаут запроса: 60 секунд.

**Вариант B (асинхронный, масштабируемый):**
- Добавить статус `in_progress` в перечисление `status`.
- Использовать очередь задач (Celery/Redis Queue или фоновый поток).
- Клиент получает немедленный ответ с идентификаторами созданных записей.
- Клиент опрашивает статус через `GET /estaff/exports` или WebSocket.

Решение зависит от ожидаемых объёмов и требований к отзывчивости интерфейса.

### 10.13. Логирование и мониторинг

Для диагностики и контроля качества интеграции:

**События для логирования:**

| Событие | Уровень | Содержимое |
|---------|---------|------------|
| Начало выгрузки | INFO | user_id, количество резюме |
| Успешная выгрузка | INFO | hh_resume_id, estaff_candidate_id, время ответа |
| Ошибка выгрузки | ERROR | hh_resume_id, код ошибки, текст (без секретов) |
| Ошибка подключения | ERROR | Тип ошибки, таймаут/сеть |
| Изменение credentials | INFO | user_id, факт изменения (без значений) |

**Метрики для мониторинга (при наличии инфраструктуры):**

- Количество выгрузок в час/сутки (success/error).
- Среднее время ответа API e-staff.
- Процент ошибок за период.
- Алерт при проценте ошибок выше порога (например, 10%).

### 10.14. Политика хранения истории

Определить срок и порядок хранения записей в `estaff_exports`:

| Вопрос | Варианты |
|--------|----------|
| Срок хранения | Бессрочно / 1 год / 3 года |
| Архивирование | Перенос старых записей в архивную таблицу |
| Удаление | Автоматическое по расписанию или вручную администратором |

Рекомендация: хранить записи бессрочно в первой версии; добавить механизм архивирования при накоплении значительного объёма (более 100 000 записей).

### 10.16. Документация для пользователей

1. Войти в панель администрирования e-staff.

**Размещение:**
- Всплывающая подсказка рядом с полями ввода в настройках.
- Ссылка на страницу справки (при наличии базы знаний).

### 10.17. Обработка специфических ошибок e-staff

Определить маппинг кодов ошибок API e-staff на сообщения пользователю:

| Код HTTP | Причина | Сообщение пользователю | Действие |
|----------|---------|------------------------|----------|
| 401 | Неверный токен | Ошибка авторизации. Проверьте настройки подключения к e-staff. | Предложить обновить credentials |
| 403 | Нет прав | Недостаточно прав для выгрузки. Обратитесь к администратору e-staff. | — |
| 404 | Вакансия не найдена | Указанная вакансия не найдена в e-staff. | Предложить выбрать другую |
| 409 | Дубликат | Кандидат уже существует в e-staff. | Предложить обновить или пропустить |
| 422 | Ошибка валидации | Некорректные данные кандидата. Проверьте заполненность резюме. | Показать детали из ответа |
| 429 | Превышен лимит | Превышен лимит запросов к e-staff. Повторите попытку позже. | Автоматический retry через интервал |
| 500, 502, 503 | Сбой сервиса | Сервис e-staff временно недоступен. Повторите попытку позже. | Автоматический retry |

Детали ошибки (тело ответа) сохранять в `error_message` для анализа службой поддержки.
