# Автоматические и ручные проверки выгрузки в e-staff

## Автоматические (pytest)

| Область | Файл / тест | Смысл |
|--------|-------------|--------|
| Базовый URL e-staff | `test_estaff_client.py` | Сборка хоста и путей |
| Маппинг HH → базовые поля | `test_estaff_client.py` — `hh_normalized_resume_to_estaff_payload` | Контакты, ФИО, зарплата |
| Справочники: разбор ответа и разрешение | `test_estaff_vocabulary.py` | `parse_get_voc_response`, `resolve_vocab_id`, нормализация |
| `get_voc` (мок) | `test_estaff_client.py` / отдельный async-тест | Успешный JSON, заголовки |
| Подготовка кандидата | `test_estaff_candidate_prepare.py` | Блоки опыта/образования, `inet_uid`, навыки при мок-справочнике |
| API выгрузки | `test_api/test_estaff.py` | Учётные данные, успех/ошибка, этап ошибки `error_stage` |
| Сообщения об ошибках e-staff | `test_estaff_client_error_message.py` | Текст для пользователя |

Запуск: из каталога `backend` при доступной тестовой БД PostgreSQL:

`pytest tests/test_estaff_client.py tests/test_estaff_vocabulary.py tests/test_estaff_candidate_prepare.py tests/test_api/test_estaff.py -q`

## Ручные сценарии

1. **Одиночная выгрузка** — из карточки кандидата: выбрана вакансия, одно резюме, успех; в ответе есть `estaff_candidate_id`, при необходимости — список `preparation_warnings`.
2. **Массовая выгрузка** — из избранного: не более 50 резюме за запрос; интерфейс блокирует превышение; итог с числом успешных и неуспешных.
3. **Ошибка получения резюме** — нет доступа к HH или неверный id: статус ошибки, `error_stage` = `fetch_resume`.
4. **Ошибка e-staff при создании** — неверные данные или конфликт: `error_stage` = `estaff_api`, сообщение пользователю без лишней технической простыни.
5. **Справочники** — регион/навык без совпадения: выгрузка может пройти без необязательных полей; предупреждения в `preparation_warnings` и на странице истории выгрузок.
6. **История выгрузок** — раздел «Выгрузки e-staff»: отображаются статусы, ошибки, идентификаторы кандидатов; не путать с «Историей» поиска по резюме.

После изменений схемы БД выполнить миграции (`alembic upgrade head`) на стенде и тестовой базе.
